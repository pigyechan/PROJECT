## 1. TDD 단위 테스트 비교

### calculateDefectRate

| 케이스 | 손으로 쓴 테스트 | 파이프라인 생성 |
|---|---|---|
| 정상 계산 | `(10, 100)` → 10.0 ✅ | `(5, 100)` → 5.0 ✅ |
| **소수점 나눗셈 함정** | `(3, 200)` → 1.5 ✅ | **없음** ❌ |
| total = 0 예외 | `(0, 0)` → 예외 ✅ | `(1, 0)` → 예외 ✅ |
| 예외 메시지 확인 | `contains("전체 수")` ✅ | `'전체 수는 1 이상이어야 합니다.'` 명시 ✅ |
| defective 음수 예외 | `(-1, 100)` → 예외 ✅ | `(-1, 10)` → 예외 ✅ |
| defective > total 예외 | `(101, 100)` → 예외 ✅ | `(11, 10)` → 예외 ✅ |
| **defective == total 허용** | `(100, 100)` → 예외 없음 ✅ | **없음** ❌ |
| 불량 0건 (최솟값) | `(0, 100)` → 0.0 ✅ | **없음** ❌ |
| 불량률 100% (최댓값) | `(100, 100)` → 100.0 ✅ | **없음** ❌ |

## 2. BDD 시나리오 비교

### 케이스 수 차이

| | 손으로 쓴 BDD | 파이프라인 생성 |
|---|---|---|
| 시나리오 수 | **16개** | **3개** |

### 손으로 쓴 시나리오 16개 전체

```gherkin
# Happy Path
Scenario: 불량률 정상 계산
  Given 전체 100개, 불량 10개 / Then 불량률 10.0%

Scenario: 소수점 불량률 계산
  Given 전체 200개, 불량 3개 / Then 불량률 1.5%

Scenario: 품질 등급 양호 판정
  Given 불량률 0.9% / Then "양호"

Scenario: 품질 등급 주의 판정
  Given 불량률 3.0% / Then "주의"

Scenario: 품질 등급 불량 판정
  Given 불량률 10.0% / Then "불량"

Scenario: DB 조회 후 불량률 계산 및 저장          ← 파이프라인 없음
  Given ID 1번에 불량 10개, 전체 100개 저장됨
  Then 불량률 10.0%와 등급 "불량"이 DB에 저장됨

Scenario: 무결점 생산 불량 0건
  Given 전체 100개, 불량 0개 / Then 불량률 0.0%

# 경계값
Scenario: 양호 상한 경계 0.9%
Scenario: 주의 시작 경계 1.0% 포함
Scenario: 주의 상한 경계 4.9%
Scenario: 불량 시작 경계 5.0% 포함
Scenario: 불량률 100% 전량 불량 허용              ← 파이프라인 없음

# 실패·거절
Scenario: 생산 수량 0이면 계산 거절
Scenario: 불량 수량 음수이면 계산 거절
Scenario: 불량 수량이 전체 초과하면 계산 거절
Scenario: 존재하지 않는 생산 실적 ID 조회         ← 파이프라인 없음
  And 결과 저장은 호출되지 않아야 한다
Scenario: 불량률 범위 벗어난 등급 판정 거절
```

### 파이프라인이 생성한 시나리오 3개

```gherkin
# ⚠️ 독립성 위반: 두 함수를 한 시나리오에서 연결
Scenario: 정상적인 불량률 계산 및 등급 판정
  Given 불량 수 2와 전체 수 100이 주어짐
  When  calculateDefectRate(2, 100)을 실행하고 결과를 evaluateGrade에 전달함
  Then  결과는 '양호' 등급이 반환됨

# ✅ 예외 메시지까지 Then에 명시
Scenario: 잘못된 입력값에 대한 예외 처리
  Given 불량 수 10과 전체 수 5가 주어짐
  When  calculateDefectRate(10, 5)를 실행함
  Then  '불량 수는 전체 수를 초과할 수 없습니다.' 메시지를 포함한
        IllegalArgumentException이 발생함

# ✅ 경계값 정확히 잡음
Scenario: 경계값(주의 등급 시작) 테스트
  Given 불량률 1.0이 주어짐
  When  evaluateGrade(1.0)을 실행함
  Then  '주의' 등급이 반환됨
```

## 3. 각자 더 잘한 것 분석

### 손으로 쓴 테스트가 더 잘한 것

**① 소수점 나눗셈 함정을 알고 있었다**

`calculateDefectRate(3, 200)` → `1.5` 케이스.

Java에서 `int / int`는 소수점을 버린다.
캐스팅 없이 `defective / total * 100`으로 구현하면 `3 / 200 = 0` → `0.0%`가 나온다.
손으로 쓴 사람은 이 함정을 TDD 과정에서 직접 만났기 때문에 테스트로 남겼다.
파이프라인은 요구사항 텍스트만 읽으므로 이 구현 함정을 알 수 없다.

**② `defective == total` 허용을 명시로 못 박았다**

`calculateDefectRate(100, 100)` → 예외 없이 100.0 반환.
이 케이스가 없으면 나중에 누군가 "100% 불량은 말이 안 된다"며 예외를 추가할 수 있다.
손으로 쓴 테스트는 그 의도를 코드로 명시했다. 파이프라인은 이 케이스를 생성하지 않았다.

**③ 경계값을 더 촘촘하게 커버했다 (BDD)**

손으로 쓴 BDD는 `0.9 / 1.0 / 4.9 / 5.0` 4개 경계를 모두 별도 시나리오로 작성했다.
파이프라인 BDD는 `1.0` 하나만 잡았다. `4.9`(주의 상한)가 빠졌다.

**④ DB와 외부 의존성이 있는 시나리오를 포함했다**

```gherkin
Scenario: DB 조회 후 불량률 계산 및 저장
Scenario: 존재하지 않는 생산 실적 ID 조회
  And 결과 저장은 호출되지 않아야 한다
```

파이프라인은 요구사항에 DB가 없었으므로 이런 시나리오를 만들 수 없었다.

---

### 파이프라인이 더 잘한 것

**① `evaluateGrade` 범위 초과 예외를 자동으로 잡았다**

손으로 쓴 단위 테스트 14개 중 `evaluateGrade(-0.1)`, `evaluateGrade(101.0)` 케이스가 없다.
구현 코드에는 이 예외 처리가 있는데 테스트로 검증하지 않았던 상태였다.
파이프라인은 요구사항을 보고 이 케이스를 자동으로 생성했다.

**② 빠르게 뼈대를 만들어준다**

아무것도 없는 상태에서 요구사항 텍스트 하나를 입력하면
단위 테스트 8개 + BDD 시나리오 3개를 수 초 안에 만들어준다.
손으로 쓰면 첫 케이스를 어디서 시작할지 고민하는 시간이 필요하다.

## 5. 결론

**파이프라인은 "빠른 초안"에 강하고, 손으로 쓴 테스트는 "맥락"에 강하다.**

파이프라인이 모르는 것:
- 구현 과정에서 만난 언어 함정 (정수 나눗셈)
- 의도적으로 허용한 설계 결정 (100% 불량 허용)
- 요구사항에 없는 외부 시스템 (DB)

파이프라인이 잘 하는 것:
- 요구사항에 명시된 예외 조건을 빠짐없이 케이스로 만들기
- 인간이 집중하다 보면 놓치기 쉬운 경계 바깥 입력 (`evaluateGrade(101.0)`)

실전에서는 **파이프라인으로 초안을 뽑고, 구현하면서 알게 된 함정과 설계 결정을 손으로 추가하는** 방식이 가장 효율적이다.
