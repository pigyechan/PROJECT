# Q2. '테스트가 충분하다'를 무엇으로 판정할 것인가

커버리지 80%를 달성한 테스트 스위트에 구멍이 있을 수 있다. 이번 경험이 그것을 직접 보여줬다.

파이프라인의 MAX_ITERATIONS는 3이다. 루브릭 점수가 임계값을 넘으면 PASS, 못 넘으면 Refine 후 재시도, 3번 안에 못 넘으면 REJECT. 이번 run은 1회에 수렴했다. 그러면 이 기준을 테스트 충분성에 그대로 적용할 수 있을까.

소프트웨어 팀이 자주 쓰는 기준이 있다. 커버리지 80%. 전체 코드 라인 중 테스트가 실행한 줄의 비율이 80%를 넘으면 충분하다고 보는 관행이다.

파이프라인이 생성한 시나리오 1번은 커버리지 도구 입장에서 이상적인 케이스다. calculateDefectRate도 실행하고 evaluateGrade도 실행한다. 두 함수 모두 커버리지에 기여한다. 그러나 아래 상황에서도 테스트는 통과한다.

```java
// calculateDefectRate를 일부러 망가뜨린 버전
public double calculateDefectRate(int defective, int total) {
    return 0.0;  // 항상 0 반환
}
```

반환값이 0.0이어도 evaluateGrade(0.0)은 "양호"를 돌려준다. 시나리오가 기대한 결과("양호")와 일치한다. 테스트 초록불이다. 커버리지도 100%다. 그러나 이 코드는 버그가 있다.

커버리지는 '실행됐다'를 측정하지, '검증됐다'를 측정하지 않는다. 80%라는 숫자는 80%의 라인이 한 번은 실행됐다는 뜻이지, 그 라인이 버그를 낼 때 테스트가 실패한다는 보장이 아니다.

충분성의 진짜 증거는 다른 질문에서 나온다. 코드를 일부러 망가뜨렸을 때 테스트가 실패하는가. 뮤테이션 테스트가 바로 그 질문에 답한다. 코드에 작은 변형(mutant)을 가한 뒤 테스트 스위트가 이를 잡아내는지를 점수로 매기는 방식이다.

**지금 당장 해볼 수 있는 것:** AI가 생성한 테스트 중 When 절이 두 함수를 연결하는 케이스를 찾아서, 첫 번째 함수를 0이나 null을 반환하도록 바꿔본다. 테스트가 여전히 통과한다면, 그 테스트는 검증하지 않는 테스트다. Java 프로젝트에서는 PIT(Pitest)로 이 과정을 자동화할 수 있다.
