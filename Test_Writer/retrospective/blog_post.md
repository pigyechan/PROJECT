# AI가 짠 테스트를 믿어도 될까

이번 Sprint에서 나는 직접 만든 도구로 테스트를 생성했다. Blog_Writer라는 글쓰기 파이프라인을 테스트 케이스 생성 파이프라인으로 이식하는 작업이었다. 결과물을 손으로 쓴 테스트와 나란히 놓고 보니, 생각보다 불편한 질문이 여러 개 남았다. 그 질문들을 그대로 정리해 두기로 했다.

---

## 아무것도 검증하지 않는 테스트는 얼마나 됐나

파이프라인은 단위 테스트 8개와 Gherkin 시나리오 3개를 생성했다. 처음 보면 그럴듯했다. 각 항목에 구체적인 값이 들어가 있었고, 예외 메시지까지 명시돼 있었다. 시니어 QA 페르소나를 부여한 Critique 단계도 빈틈 3개를 정확히 짚어냈다.

그런데 시나리오 1번을 다시 읽었을 때 멈칫했다.

```gherkin
Given  불량 수 2와 전체 수 100이 주어짐
When   calculateDefectRate(2, 100)을 실행하고 결과를 evaluateGrade에 전달함
Then   결과는 '양호' 등급이 반환됨
```

두 함수를 한 시나리오에서 연결했다. calculateDefectRate가 2.0을 반환하면 evaluateGrade는 '양호'를 돌려준다. 여기까지는 맞다. 그런데 만약 calculateDefectRate를 일부러 망가뜨려 항상 0.0을 반환하도록 바꾸면 어떻게 될까. 0.0은 1.0 미만이므로 evaluateGrade는 여전히 '양호'를 반환한다. 테스트는 초록불 그대로다.

이 테스트는 calculateDefectRate가 옳은지 검증하지 않는다. 두 함수가 연결되어 어떤 결과를 냈는지만 확인한다. 코드를 망가뜨려도 통과하는 테스트 — 11개 중 적어도 1개, 비율로는 약 9%가 여기에 해당했다.

더 당혹스러운 건 이 부분이었다. Critique는 independence 문제를 정확하게 짚었다. 그런데 Eval이 independence 점수로 5.0 만점을 줬다. 왜냐하면 Eval은 Critique 결과를 받지 않기 때문이다. Gen/Eval 격리 원칙의 부작용이다. 파이프라인 구조가 옳아도 결과가 틀릴 수 있다는 것을 이번에 직접 목격했다.

---

## '테스트가 충분하다'는 기준은 무엇인가

파이프라인의 MAX_ITERATIONS는 3이다. 루브릭 점수가 임계값을 넘으면 PASS, 못 넘으면 Refine 후 재시도, 3번 안에 못 넘으면 REJECT. 이번 run은 1회에 수렴했다. 그러면 이 기준을 테스트 충분성에 그대로 적용할 수 있을까.

소프트웨어 팀이 테스트 충분성의 기준으로 자주 쓰는 수치가 있다. 커버리지 80%. 전체 코드 라인 중 테스트가 실행한 줄의 비율이 80%를 넘으면 충분하다고 보는 관행이다.

그런데 이번 경험이 그 기준에 정면으로 의문을 던진다. 파이프라인이 생성한 시나리오 1번은 커버리지 측정 도구 입장에서는 완벽하다. calculateDefectRate도 실행하고 evaluateGrade도 실행한다. 두 함수 모두 커버리지에 기여한다. 그러나 calculateDefectRate가 잘못된 값을 반환해도 테스트는 통과한다.

커버리지는 '실행됐다'를 측정하지, '검증됐다'를 측정하지 않는다. 80%라는 숫자는 80%의 라인이 한 번은 실행됐다는 뜻이지, 그 라인이 버그를 낼 때 테스트가 실패한다는 보장이 아니다.

충분성의 진짜 증거는 다른 질문에서 나온다. 코드를 일부러 망가뜨렸을 때 테스트가 실패하는가. 뮤테이션 테스트가 바로 그 질문에 답한다. 테스트 스위트가 돌연변이 코드를 얼마나 잡아내는지를 점수로 매기는 방식이다. 이번 Sprint에서 AI 생성 테스트의 허점을 직접 목격했기 때문에, 다음 단계로 뮤테이션 테스트가 자연스럽게 이어진다.

---

## 같은 버그를 어느 쪽이 먼저 잡았나

비교 결과는 생각보다 명확했다. 인간이 먼저 잡은 버그가 있었고, AI가 먼저 발견한 구멍도 있었다.

**인간이 먼저 잡은 것.** Java의 정수 나눗셈 함정이다. `calculateDefectRate(3, 200)`은 1.5를 반환해야 하는데, `int / int`로 계산하면 소수점이 버려져 0이 나온다. `(double) defective / total`로 캐스팅하지 않으면 버그가 생긴다. TDD를 직접 하면서 이 함정을 만났기 때문에 테스트로 남긴 것이다. 파이프라인은 요구사항 텍스트만 읽었다. 텍스트에는 "불량률(%)을 반환한다"고 적혀 있을 뿐, Java 정수 나눗셈의 함정은 적혀 있지 않다. 알 수 없다.

`calculateDefectRate(100, 100) → 100.0, 예외 없음`도 그렇다. 100% 불량이 실제로 발생 가능한 상황이라 예외를 던지지 않아야 한다는 설계 결정이다. 이 의도를 테스트로 못 박지 않으면 나중에 누군가 "이건 말이 안 된다"며 예외를 추가할 수 있다. 파이프라인은 이 케이스를 생성하지 않았다.

**AI가 먼저 잡은 것.** `evaluateGrade`에 음수나 100 초과 값을 넣을 때의 예외 처리다. 구현 코드에는 이 로직이 있었다. `if (defectRate < 0 || defectRate > 100) throw ...`. 그런데 손으로 쓴 단위 테스트 14개 중에 이 경우를 검증하는 케이스가 없었다. 파이프라인은 요구사항에서 그 조건을 읽고 `evaluateGrade(-0.1)`, `evaluateGrade(101.0)` 케이스를 자동으로 만들었다.

이 경험이 말하는 역할 분담은 이렇다. AI는 요구사항에 명시된 조건을 빠짐없이 케이스로 변환하는 데 빠르고 꼼꼼하다. 인간은 구현 과정에서 만난 언어 함정, 의도적인 설계 결정, 비즈니스 맥락을 케이스에 담을 수 있다. 둘은 경쟁 관계가 아니라 서로 다른 사각지대를 메우는 관계다. 파이프라인으로 초안을 뽑고, 구현하면서 만난 맥락을 인간이 추가하는 방식이 현실적인 협업 모델이다.

## 인수테스트가 AI 시대에 하는 역할

이번 Sprint 내내 하나의 대비가 마음에 걸렸다.

`validate.py`는 결정적이다. JSON 스키마를 검사하고, Then 절에 "정상 동작"이라는 표현이 있으면 REJECT를 낸다. 같은 입력이 들어오면 항상 같은 판정이 나온다. 이것은 계약이다.

rubric.yaml 기반 Eval은 비결정적이다. Blog_Writer를 실행할 때 같은 글에 대해 weighted_total 3.15가 나오기도 하고 3.8이 나오기도 하고 3.35가 나오기도 했다. LLM이 매번 조금씩 다르게 채점하기 때문이다. 이것은 의견이다.

두 가지가 파이프라인 안에 공존한다. Eval은 유연하게 "이 테스트가 충분히 좋은가"를 판단하지만 그 판단은 흔들린다. validate.py는 흔들리지 않는다. "Then 절에 모호한 표현이 있으면 무조건 REJECT"는 어떤 모델이 채점해도 바뀌지 않는 규칙이다.

이 경험을 AI가 짠 코드를 판정하는 상황으로 확장하면 이야기가 정확히 겹쳐진다.

AI가 코드를 생성하는 시대에 코드 리뷰는 점점 LLM이 대신하게 된다. "이 코드가 좋은 코드인가"를 LLM에게 물으면 매번 조금씩 다른 의견이 돌아온다. 3.15점이 나오기도 하고 3.8점이 나오기도 한다. 유연하지만 믿기 어렵다.

반면 인수테스트는 validate.py와 같다. 시나리오를 실행해서 기대한 동작이 나오면 GREEN, 아니면 RED다. 같은 코드를 백 번 실행해도 판정이 바뀌지 않는다. AI가 생성한 코드가 요구사항을 실제로 만족하는지를 판정하는 유일한 결정적 수단이다.

Gherkin 시나리오가 그래서 중요하다. 자연어에 가까운 형식으로 요구사항을 기술하면서도 자동으로 실행 가능하다. AI가 코드를 생성한 뒤 인수테스트를 돌려 GREEN이 나오면 "이 코드는 요구사항을 만족한다"고 결정적으로 말할 수 있다. LLM 채점처럼 "아마도 좋은 코드일 것이다"가 아니라.

이번 Sprint에서 Critique가 독립성 문제를 정확히 지적했는데도 Eval이 5.0 만점을 줬던 장면이 기억에 남는다. AI 의견은 흔들렸고, 규칙은 흔들리지 않았다. 다음 Sprint에서 AI가 짠 코드를 검증하는 인수테스트를 설계할 때, 나는 그 장면을 기준으로 삼을 것이다. LLM 채점이 아닌 인수테스트로 판정한다. 의견이 아닌 계약으로.

---

## 도구를 직접 써보며 배운 것

마지막으로 하나를 더 남겨 둔다. 이번 Sprint에서 내가 만든 파이프라인을 내 글쓰기에 써봤다. Blog_Writer를 돌려서 이 글의 초안을 다듬었다. 도구를 만드는 것과 도구를 쓰는 것은 다른 경험이다.

만들 때는 파이프라인이 어떻게 돌아가는지를 생각한다. 쓸 때는 결과물이 충분한지를 생각한다. 이번 Sprint에서 내가 목격한 것은 바로 그 간극이다. 파이프라인은 잘 돌아갔다. 1회에 수렴했고 PASS를 받았다. 그런데 결과물에는 아무것도 검증하지 않는 테스트가 섞여 있었다.

좋은 도구는 "잘 돌아간다"와 "좋은 결과를 낸다"를 동시에 만족해야 한다. 지금 파이프라인은 전자는 됐고 후자는 아직 부족하다. 뮤테이션 테스트가 다음 Sprint의 씨앗이 된 이유가 여기 있다.
