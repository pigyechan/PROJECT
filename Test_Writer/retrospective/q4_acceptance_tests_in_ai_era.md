# Q4. 인수테스트가 AI 시대 개발에서 어떤 역할을 하는가

AI가 코드를 생성하는 시대에, 그 코드가 요구사항을 만족하는지를 판정하는 유일한 결정적 수단이 있다. 인수테스트다. 이번 Sprint에서 파이프라인을 직접 만들면서 그 이유를 이해하게 됐다.

이번 Sprint 내내 하나의 대비가 마음에 걸렸다.

`validate.py`는 결정적이다. JSON 스키마를 검사하고, Then 절에 "정상 동작"이라는 표현이 있으면 REJECT를 낸다. 같은 입력이 들어오면 항상 같은 판정이 나온다. 이것은 계약이다.

rubric.yaml 기반 Eval은 비결정적이다. Blog_Writer를 실행할 때 같은 글에 대해 weighted_total 3.15가 나오기도 하고 3.8이 나오기도 하고 3.35가 나오기도 했다. LLM이 매번 조금씩 다르게 채점하기 때문이다. 이것은 의견이다.

두 가지가 파이프라인 안에 공존한다. Eval은 유연하게 "이 테스트가 충분히 좋은가"를 판단하지만 그 판단은 흔들린다. validate.py는 흔들리지 않는다. "Then 절에 모호한 표현이 있으면 예외 없이 REJECT"는 어떤 모델이 채점해도 바뀌지 않는 규칙이다.

이 경험을 AI가 짠 코드를 판정하는 상황으로 확장하면 이야기가 정확히 겹쳐진다.

AI가 코드를 생성하면 LLM 기반 코드 리뷰가 점점 일반화된다. "이 코드가 좋은 코드인가"를 LLM에게 물으면 매번 조금씩 다른 의견이 돌아온다. 3.15점이 나오기도 하고 3.8점이 나오기도 한다. 유연하지만 믿기 어렵다.

반면 인수테스트는 validate.py와 같다. 시나리오를 실행해서 기대한 동작이 나오면 PASS, 아니면 FAIL이다. 같은 코드를 백 번 실행해도 판정이 바뀌지 않는다. AI가 생성한 코드가 요구사항을 실제로 만족하는지를 판정하는 결정적 수단이다.

Gherkin 시나리오가 그래서 중요하다. 자연어에 가까운 형식으로 요구사항을 기술하면서도 자동으로 실행 가능하다. AI가 코드를 생성한 뒤 인수테스트를 돌려 GREEN이 나오면 "이 코드는 요구사항을 만족한다"고 결정적으로 말할 수 있다. LLM 채점처럼 "아마도 좋은 코드일 것이다"가 아니라.

이번 Sprint에서 Critique가 독립성 문제를 정확히 지적했는데도 Eval이 5.0 만점을 줬던 장면이 기억에 남는다. AI 의견은 흔들렸고, 규칙은 흔들리지 않았다.

**지금 바로 적용할 수 있는 것:** AI로 코드를 생성할 때, LLM에게 "이 코드 괜찮아?"라고 묻는 대신 Gherkin 시나리오를 먼저 작성한다. 시나리오를 통과하면 합격, 통과하지 못하면 재생성이나 수정을 요청한다. LLM 의견이 아닌 인수테스트로 판정한다. 의견이 아닌 계약으로.
