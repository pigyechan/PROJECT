# Q1. AI가 짠 테스트 중 '항상 통과하는 테스트'의 비율은

코드를 일부러 망가뜨렸는데 테스트가 초록불이라면, 그 테스트는 존재하지 않는 것과 같다. 이번에 AI가 생성한 11개 테스트 중 그런 케이스가 있었다.

파이프라인은 단위 테스트 8개와 Gherkin 시나리오 3개를 생성했다. 처음 보면 그럴듯했다. 각 항목에 구체적인 값이 들어가 있었고, 예외 메시지까지 명시돼 있었다. 그런데 시나리오 1번을 다시 읽었을 때 멈칫했다.

```gherkin
Given  불량 수 2와 전체 수 100이 주어짐
When   calculateDefectRate(2, 100)을 실행하고 결과를 evaluateGrade에 전달함
Then   결과는 '양호' 등급이 반환됨
```

두 함수를 한 시나리오에서 연결했다. 문제는 여기서 나온다. calculateDefectRate를 일부러 망가뜨려 항상 0.0을 반환하도록 바꾸면 어떻게 될까. 0.0은 1.0 미만이므로 evaluateGrade는 여전히 '양호'를 반환한다. 테스트는 초록불 그대로다.

이 테스트는 calculateDefectRate가 옳은지 검증하지 않는다. 두 함수가 연결되어 어떤 결과를 냈는지만 확인한다. **코드를 망가뜨려도 통과하는 테스트 — 11개 중 적어도 1개, 비율로는 약 9%가 여기에 해당했다.**

더 당혹스러운 건 따로 있었다. Critique는 이 독립성 문제를 정확히 짚었다. 그런데 Eval이 independence 점수로 5.0 만점을 줬다. Eval은 Critique 결과를 받지 않기 때문이다. Gen/Eval 격리 원칙의 부작용이다. AI 파이프라인 구조가 옳아도 결과가 틀릴 수 있다는 것을 직접 목격했다.

**AI가 만든 테스트를 신뢰하기 전에 할 것:** 각 테스트가 검증하는 함수를 하나로 제한하는지 직접 확인한다. When 절에 두 함수가 연결돼 있다면 독립성을 의심하고, 그 함수만 망가뜨렸을 때 테스트가 실패하는지 수동으로 확인해 본다. 이것이 다음 Sprint의 뮤테이션 테스트가 자동화하려는 검증이다.
