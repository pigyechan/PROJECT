# Q3. 같은 버그를 어느 쪽이 먼저 잡았나 — 인간과 AI의 역할 분담

비교 결과는 생각보다 명확했다. 인간이 먼저 잡은 버그가 있었고, AI가 먼저 발견한 구멍도 있었다. 어느 쪽이 더 뛰어난지가 아니라, 각자 보는 것이 다르다는 게 핵심이다.

**인간이 먼저 잡은 것.** Java의 정수 나눗셈 함정이다. `calculateDefectRate(3, 200)`은 1.5를 반환해야 하는데, `int / int`로 계산하면 소수점이 버려져 0이 나온다. `(double) defective / total`로 캐스팅하지 않으면 버그가 생긴다. TDD를 직접 하면서 이 함정을 만났기 때문에 테스트로 남긴 것이다. 파이프라인은 요구사항 텍스트만 읽었다. 텍스트에는 "불량률(%)을 반환한다"고 적혀 있을 뿐, Java 정수 나눗셈의 함정은 적혀 있지 않다. 알 수 없다.

`calculateDefectRate(100, 100) → 100.0, 예외 없음`도 그렇다. 100% 불량이 실제로 발생 가능한 상황이라 예외를 던지지 않아야 한다는 설계 결정이다. 이 의도를 테스트로 못 박지 않으면 나중에 누군가 "이건 말이 안 된다"며 예외를 추가할 수 있다. 파이프라인은 이 케이스를 생성하지 않았다.

**AI가 먼저 잡은 것.** `evaluateGrade`에 음수나 100 초과 값을 넣을 때의 예외 처리다. 구현 코드에는 이 로직이 있었다.

```java
if (defectRate < 0 || defectRate > 100) {
    throw new IllegalArgumentException("불량률은 0 이상 100 이하여야 합니다.");
}
```

그런데 손으로 쓴 단위 테스트 14개 중에 이 경우를 검증하는 케이스가 없었다. `evaluateGrade(-0.1)`, `evaluateGrade(101.0)` — 구현 코드에 있는 예외 처리인데 테스트가 없었다. 파이프라인은 요구사항에서 그 조건을 읽고 자동으로 케이스를 만들었다.

이 경험이 말하는 역할 분담은 이렇다. AI는 요구사항에 명시된 조건을 빠짐없이 케이스로 변환하는 데 빠르고 꼼꼼하다. 인간은 구현 과정에서 만난 언어 함정, 의도적인 설계 결정, 비즈니스 맥락을 케이스에 담을 수 있다. 둘은 경쟁 관계가 아니라 서로 다른 사각지대를 메우는 관계다.

**실전에서 쓸 수 있는 워크플로우:** 파이프라인으로 요구사항 기반 초안을 뽑는다(10분). 그 다음 구현하면서 만난 언어 함정, 허용하기로 결정한 경계 케이스를 직접 추가한다(20분). 두 번째 단계가 첫 번째 단계보다 짧아진다는 게 이 방식의 가치다.
