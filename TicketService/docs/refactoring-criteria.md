# 리팩토링이 필요한 경우 vs 불필요한 경우 — 본인 기준

## 판단 도구

1. **Fowler 코드 스멜 카탈로그**로 "이 코드에 이름 붙은 문제가 있는가"를 먼저 확인한다.
   이번 kata의 원본 `TicketService.reserveTicket`은 최소 두 가지 스멜이 있었다.
   - Anemic Ticket → **Feature Envy** (TicketService가 Ticket의 필드를 훔쳐보고 대신 판단)
   - 유저 조회+티켓 검증+결제+영속화가 한 메서드 → **Long Method** / **Divergent Change**
     (결제 방식이 바뀌는 이유와 예약 규칙이 바뀌는 이유가 같은 메서드에 섞여 있음)
2. 스멜이 있다고 무조건 리팩토링하지 않는다. **비용/효용 저울**로 한 번 더 거른다.
   - 비용: 안전망(테스트) 있는가, 팀이 이 코드 구조를 이해하는 데 드는 시간, 리팩토링 자체가
     버그를 심을 위험
   - 효용: 이 코드를 **앞으로 얼마나 자주, 어떤 이유로** 다시 열게 될 것인가

## 필요한 경우

- 스멜이 있고, **변경이 예정돼 있거나 이미 반복되고 있음** (결제수단 추가, 알림 채널 추가 등
  "다음 요구사항"이 뻔히 보일 때)
- 그 스멜 때문에 **버그가 실제로 발생했거나 발생 직전** (예: 결제 실패했는데 예약 상태가
  바뀌는 순서 실수 — 이번 kata에서 Move Method 단계 때 실제로 순서를 한 번 깨뜨릴 뻔했다)
- **안전망(특성화 테스트)을 먼저 깔 수 있는** 코드다. 안전망 없이 구조를 바꾸는 건 리팩토링이
  아니라 도박이다.

## 불필요한 경우

- 스멜은 있지만 **그 코드를 다시 열 계획이 없다** (수명이 끝나가는 배치 스크립트, 곧 폐기될
  기능). 안 열 코드를 예쁘게 만드는 건 효용이 0에 수렴한다.
- **테스트도 없고, 도메인 지식을 가진 사람도 없는데** 마감이 급한 경우. 이럴 땐 리팩토링보다
  차라리 스멜을 그대로 두고 옆에 새 코드를 격리해서 추가하는 쪽이 더 안전하다 (Strangler Fig에
  가까운 선택).
- 로직 자체가 없는 순수 데이터 객체(B-3에서 다룬 DTO, Read Model)에 "혹시 몰라서" 행위를
  미리 만들어두는 경우 — 지킬 규칙이 없는데 그릇만 만드는 과잉설계.

## 멘토 프레임: 재즈 3단계

- **모방**: 이번 kata에서 Fowler 카탈로그의 기법 이름(Move Method, Extract Interface, Rename
  Class, Extract Method, Remove Setting Method)을 그대로 하나씩 순서대로 적용했다.
- **응용**: Move Method 단계에서 원본 동작(결제 전 중복예약 차단)을 지키기 위해 `isReserved()`
  가드를 `TicketService`에 한 단계 더 남겨두는 절충을 뒀다 — 카탈로그가 "정답"을 주지 않는
  지점에서 캐릭터라이제이션 테스트가 지켜야 할 계약을 우선했다.
- **자기화**: 이 문서의 "불필요한 경우" 섹션 자체가 자기화다. 기법을 아는 것과, 지금 이 코드에
  기법을 **쓰지 않기로** 결정하는 것은 다른 능력이다. 스멜이 보인다고 반사적으로 리팩토링하는 건
  아직 모방 단계에 머무는 것이고, "이 스멜은 지금 건드릴 가치가 없다"고 말할 수 있어야 응용을
  넘어선 것이다.
